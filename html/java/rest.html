RESTful架构

作者：吴杨杨 返回首页

时间：2016-07-31

参考资料：http://www.ruanyifeng.com/blog/2011/09/restful

REST 不是一种具体的技术，也不是一种具体的规范，它是一种设计的风格。

REST指的是一组架构约束条件和原则。满足这些约束条件和原则的应用程序或设计就是 RESTful。

随着 REST 成为大多数 Web 和 Mobile 应用的默认选择，势必要对它的基本原理有所了解。在它提出十多年后的今天，REST 已经成为最重要的 Web 应用技术之一。随着所有技术朝着 API 方向发展，它的重要性有可能持续快速地增长,Web 开发者和架构师对 REST 和 RESTful 服务有一个清晰的理解是很重要的。

REST 描述了一个架构样式的互联系统（如 Web 应用程序）。REST 约束条件作为一个整体应用时，将生成一个简单、可扩展、有效、安全、可靠的架构。开发人员可以轻松使用 Ajax 和 RESTful Web 服务一起创建丰富的界面。

REST（英文：Representational State Transfer，简称REST）首次出现在 2000 年 Roy Fielding 的博士论文中，他是 HTTP 规范的主要编写者之一。阮一峰老师翻译为"表现层状态转化"，我觉得很形象。

"表现层"其实指的是"资源"（Resources）的"表现层"。我们把"资源"具体呈现出来的形式，叫做它的"表现层"（Representation）。任何事物，只要有被引用到的必要，它就是一个资源。它可以是一首歌、一篇文章。你可以用一个URL指向它，每种资源对应一个特定的URL。

URI的设计应该遵循可寻址性原则，具有自描述性，要让URL作者以外的人能明白这个URL是干什么的。

因为"资源"表示一种实体,所以应该是名词,URL不应该有动词,动词应该放在HTTP协议中。

互联网通信协议HTTP协议，是一个无状态协议，因此，如果客户端想要操作服务器，必须通过某种手段，让服务器端发生"状态转化"（State Transfer）。而这种转化是建立在表现层之上的，所以就是"表现层状态转化"。

客户端用到的手段，只能是HTTP协议。具体来说，就是HTTP协议里面，四个表示操作方式的动词：GET、POST、PUT、DELETE。

  • GET (select) 获取资源
  • PUT (update) 更新资源
  • POST (insert) 新增资源
  • DELETE (delete) 删除资源

下面，我们来看一些 RESTful API 例子：

  • GET /user 获取所有用户信息
  • PUT /user/ID 根据用户ID更新某个用户的信息
  • POST /user 新增一个用户信息
  • DELETE /user/ID 根据用户ID删除某个用户的信息

因为 RESTful API 不是具体的技术和规范，所以，我们再来找找感觉：

  • GET /department 获取所有部门信息
  • GET /department/ID 获取某个部门的信息
  • GET /department/ID/employee 获取某个部门的所有员工信息
  • GET /department/ID/employee/ID 获取某个部门的某个员工的信息
  • PUT /department 更新所有部门信息
  • PUT /department/ID 更新某个部门的信息
  • PUT /department/ID/employee 更新某个部门的所有员工信息
  • PUT /department/ID/employee/ID 更新某个部门的某个员工的信息
  • POST /department 新增一个部门
  • POST /department/ID/employee 某个部门新增一个员工
  • DELETE /department 删除所有部门
  • DELETE /department/ID 删除某个部门
  • DELETE /department/ID/employee 删除某个部门的所有员工
  • DELETE /department/ID/employee/ID 删除某个部门的某个员工

如果记录数量很多，服务器不可能都将它们返回给用户。API应该提供参数，过滤返回结果。例如：

  • ?limit=10：指定返回记录的数量
  • ?page=2&per_page=100：指定第几页，以及每页的记录数。

以上。
